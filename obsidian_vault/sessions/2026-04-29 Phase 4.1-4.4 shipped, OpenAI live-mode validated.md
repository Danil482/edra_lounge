# 2026-04-29 Phase 4.1 → 4.4 — OpenAI provider + real LinkedIn validated end-to-end

Длинная сессия с двумя сменами провайдера, переписанным парсером, новым LLM-провайдером, отлаженными промптами и видимым логированием. Booth теперь boot'ится в live-режиме с OpenAI gpt-4o-mini против реального LinkedIn-профиля и проходит полную 5-турновую сессию с персонализированным опенером и continuation'ами.

## Хронология (по коммитам)

### Phase 4 — Live-mode default (`04d43a5`, до этой сессии)
Уже было — flip `live_mode: bool = True` в config + `RAPIDAPI_KEY=mock` дефолт.

### Phase 4.1 — Switch to linkedin-data-api 2-endpoint flow + disk cache (`c533f84`)
Старый провайдер `fresh-linkedin-profile-data.p.rapidapi.com` имел 1 эндпоинт с постами в том же ответе. Юзер захотел провайдера с 2 раздельными эндпоинтами (`/get-profile-data-by-url` + `/get-profile-posts?username=`). Переписал `LinkedInRapidAPISource` под этот flow + добавил on-disk cache в `data/linkedin_cache/<slug>.json` чтобы повторный фетч того же URL не сжигал 50/мес квоту. Cache opt-in через `cache_dir` constructor arg (None в тестах). TTL plumbing заложен (`cache_ttl_seconds`), пока None — поставим 86400 ближе к booth-event'у.

### Phase 4.2 — OpenAI provider + switch RapidAPI to fresh-linkedin-scraper-api (`fd238c7`)
**Двойная смена провайдера в одной фазе**:
- LLM: добавил OpenAI как третий `LLM_MODE` (`local`/`remote` остались). Chat Completions API через httpx, без SDK. Поля `openai_api_key` / `openai_model` в Settings, новые `_openai_complete` / `_openai_stream` ветки в `llm/client.py`. Anthropic недоступен из РФ без VPN, OpenAI выбран как simplest-to-use вариант для dev (потом swap на Claude когда настроим)
- RapidAPI: первый провайдер `linkedin-data-api.p.rapidapi.com` ответил HTTP 451 *(US embargo, нужен VPN)*. Включили VPN — пришёл HTTP 200 с `{"success": false, "message": "We are no longer providing this service at this time"}`. Попробовали `linkedin-api8.p.rapidapi.com` (тот же провайдер, другой DNS) — то же sunset-сообщение. Юзер нашёл `fresh-linkedin-scraper-api.p.rapidapi.com` (saleleads.ai docs) — переписал парсер целиком под их форму
- **Hybrid input**: identifier теперь принимает либо URL, либо просто handle (`danil-onishchenko-30876037a`), нормализуется в каноничный URL для cache-key и source_identifier. Пользователю booth не надо вспоминать формат
- **Parser hardening после столкновения с реальностью**:
  - `experiences` приходит как `{total, has_more, data: [...]}` обёртка, не голый list
  - `date.start/end` — строки `"Feb 2026"` / `"Present"`, не `{year: N}` (доки наврали)
  - `bio: null` даже с `include_bio=true` (если пользователь не заполнил About)
  - HTML 200 с `success: false` нужно surface как `ProfileSourceUnavailable` (а не сваливать в "missing first_name")
  - Тело HTTP-ошибки теперь в exception message — RapidAPI'шный текст видно сразу
- **Dev-aid**: `data/linkedin_raw/` — сырые ответы дампятся туда после каждого успешного фетча, чтобы итерировать парсер без сжигания квоты. Gitignored
- **Mock-ветка** короткозамыкает раньше парсинга identifier'а — booth с `RAPIDAPI_KEY=mock` всегда показывает захардкоженный профиль автора, независимо от ввода

### Phase 4.3 — Opener prompt + LinkedIn avatar plumbing (`5eb9970`)
**Промпт**: после первого живого запуска юзер увидел "I noticed your X — what surprised you?" Открытый вопрос — а у визитёра только 3 кнопки (`Tell me more` / `Skeptical, why Defy?` / `Not interested`). Переписал `prompts/opener.txt`:
- Цитирую дословно лейблы кнопок
- Запрещаю open-ended questions в адрес визитёра
- `opener_type=question` редизайнил как *rhetorical* yes/no setup, который сразу пивотит в утверждение
- Дал good/bad shape examples в самом промпте

**Avatar**: добавил `Profile.avatar_url: str | None`, колонку `profiles.avatar_url Text` + idempotent ALTER-TABLE миграция в `db.init_db()` (catch duplicate-column-name, no Alembic). Парсер выбирает 200×200 → 400 → 800 → 100 → first из `data.avatar[]`. Frontend кладёт `<img class="profile-portrait-img">` в существующий portrait-box, реальное изображение reveal'ится только после `onload` (silent fallback на placeholder если signed URL 404'ит). CSS `filter: grayscale(1) contrast(1.05)` чтобы вписаться в documentary B&W стиль панели.

### Phase 4.4 — LLM-driven continuations + visible logging (`d70aeda`)
Юзер: "после двух Tell me more диалог перестает выводить новые реплики; в логах ничего не вижу". **Две причины**:

1. **Continuation турны вообще не шли через LLM** — `templates.render_continuation` имел 5 хардкодных шаблонов (по tone), повторение неизбежно. Шаблоны ещё и open questions содержали (`"What signal would tell you this isn't that?"`). Переписал `_produce_continuation` через LLM с новым `prompts/continuation.txt`: даёт LLM полную dialogue history (формат `Agent (turn N): ... / Visitor: [Tell me more.]`), требует не повторяться, ветви по `last_choice` (positive → advance с новой деталью, skeptical → defuse с конкретной причиной про Defy, negative → graceful close). Templates остались как fallback при offline-LLM, но переписаны на statements/narrow-yes-no — без open questions
2. **Логирование не было сконфигурено** — нигде не было `basicConfig`, поэтому `log.info` от модулей молча падали в void. Добавил `logging.basicConfig(level=INFO, format=...)` в `app.py` + per-call логи в `llm/client.py` (`LLM call → ... (prompt=N chars)` + `LLM call ✓/✗ ... in Nms`)

### Phase 4.4.1 — Verbose diagnostic logging (`3491146`)
Когда юзер прислал console-output с одним `setVisitorAvatar: no url, showing placeholder` без серверных логов — добавил:
- `_avatar_url_from_data` логирует выбранный URL (или почему None)
- `take_turn` логирует каждый turn (id/turn#/choice/interest/delta) и каждый next-step (id/turn#/reply preview)
- Frontend `setVisitorAvatar` console.log/warn'ит loading/loaded/failed

### Phase 4.4.2 — `session.profile-fetched` log (`5ff5ec4`)
Один INFO-line при `start_session` показывает `Profile.avatar_url` сразу после fetch. Если в логе видно URL но фронт говорит "no url" → баг в serialization/frontend, а не в парсере. Если в логе `avatar_url=None` → парсер виноват.

## Ключевые архитектурные решения

- **No SDK для LLM** — все три провайдера (Ollama/Anthropic/OpenAI) через `httpx` напрямую (TASK.md §3). OpenAI streaming это SSE с `data: [DONE]` sentinel + `delta.content` поля
- **Hybrid identifier input** — нормализация в одном месте (`_username_from_input`), внутренне всегда canonical URL form. Если завтра менять провайдера на тот что хочет URL — фронту менять ничего не надо
- **Mock-branch до парсинга** — `RAPIDAPI_KEY=mock` короткозамыкает _до_ извлечения handle, потому что booth-демка должна показывать стабильный пример профиля независимо от ввода
- **Disk cache stores raw API response, not Profile object** — это позволило фиксить парсер (Phase 4.2 wrap-bug) и автоматически получать обновлённые Profile при следующем чтении кеша, без re-fetch'а
- **Idempotent migration в init_db** — без Alembic, просто `try: ALTER TABLE ADD COLUMN; except: pass`. Подходит пока schema-changes редкие и single-dev
- **Templates как fallback, не основной путь** — Phase 1B было наоборот. Теперь LLM по умолчанию, шаблоны только когда LLM offline

## Открытые вопросы / следующие шаги

См. `current priorities.md`. Кратко:
1. **Continuation diversity** — все ответы LLM в одном русле ("ещё одна credential": партнёрство, cohort, case studies, scoping call). Нужна ротация типов (вопрос → личное → история → call to action) через явные категории в промпте
2. **Scenario testing** — пока тестировали только positive×5. Прогнать реальные ветки skeptical / negative / mixed, посмотреть качество ответов. Проверить как идёт graceful close на negative
3. **Avatar caching strategy** — signed URLs от LinkedIn живут несколько месяцев, потом 404. Сейчас cache хранит URL forever. Стоит либо парсить `e=` query param и invalidate'ить cache при истечении, либо проксировать аватары через бэк
4. **Profile classification для live** — `cluster_id: —` в правой панели для live-профилей, потому что embedding-классификация не реализована. Это блок к persona-based rules для real LinkedIn визитёров (см. ниже)
5. **2-уровневая кластеризация (отложенный диалог с юзером)** — кластеризуем episode-summaries для induction, но profile-cluster ищем тем же эмбеддингом. Возможно нужен раздельный profile-embedding для live-классификации

## Что узнал нового про экосистему

- **RapidAPI providers очень нестабильны** — два провайдера sunset'нулись за неделю. Disk cache + raw dump = страховка от повторного парсера-rewrite'а
- **OpenAI gpt-4o-mini хорош для continuation** — 2-3s latency, ~$0.0001/turn, варьирует ответы естественно
- **PowerShell кавычки** — для curl нужен `--%` stop-parsing token или Invoke-RestMethod с hashtable→ConvertTo-Json. JSON в одинарных кавычках в PS не работает
- **Windows console encoding** — мои `→` / `✓` / `✗` рендерятся как `→` / `✓` / `✗` в логах PowerShell, но это decode-side артефакт, не баг
- **`python-dotenv` strip'ает кавычки** автоматом, но multi-line strings (Python tuple-style) — нет, нужно одной строкой
