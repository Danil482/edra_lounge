## Timeline (by commit)

### Phase 4 — Live-mode default (`04d43a5`, before this session)

### Phase 4.1 — Switch to linkedin-data-api 2-endpoint flow + disk cache (`c533f84`)
The old provider `fresh-linkedin-profile-data.p.rapidapi.com` had a single endpoint with posts in the same response. The user wanted a provider with 2 separate endpoints (`/get-profile-data-by-url` + `/get-profile-posts?username=`). Rewrote `LinkedInRapidAPISource` to this flow and added an on-disk cache at `data/linkedin_cache/<slug>.json` so that re-fetching the same URL does not burn the 50/month quota. The cache is opt-in via the `cache_dir` constructor arg (None in tests). TTL plumbing is in place (`cache_ttl_seconds`), currently None — will be set to 86400 closer to the booth event.

### Phase 4.2 — OpenAI provider + switch RapidAPI to fresh-linkedin-scraper-api (`fd238c7`)
**Double provider swap in one phase**:
- LLM: added OpenAI as a third `LLM_MODE` (`local`/`remote` remain). Chat Completions API via httpx, no SDK. Fields `openai_api_key` / `openai_model` in Settings, new `_openai_complete` / `_openai_stream` branches in `llm/client.py`. Anthropic is unavailable from Russia without a VPN, so OpenAI was picked as the simplest-to-use option for dev (will swap to Claude later when we configure that)
- RapidAPI: the first provider `linkedin-data-api.p.rapidapi.com` answered HTTP 451 *(US embargo, needs VPN)*. With a VPN on, we got HTTP 200 with `{"success": false, "message": "We are no longer providing this service at this time"}`. We tried `linkedin-api8.p.rapidapi.com` (same provider, different DNS) — same sunset message. The user found `fresh-linkedin-scraper-api.p.rapidapi.com` (saleleads.ai docs) — rewrote the parser entirely against their shape
- **Hybrid input**: the identifier now accepts either a URL or just a handle (`danil-onishchenko-30876037a`), normalised into a canonical URL for the cache key and `source_identifier`. The booth user does not have to remember the format
- **Parser hardening after meeting reality**:
  - `experiences` arrives as `{total, has_more, data: [...]}` wrapper, not a bare list
  - `date.start/end` are strings like `"Feb 2026"` / `"Present"`, not `{year: N}` (the docs lied)
  - `bio: null` even with `include_bio=true` (if the user did not fill in About)
  - HTTP 200 with `success: false` must be surfaced as `ProfileSourceUnavailable` (not fall through to "missing first_name")
  - The HTTP error body now goes into the exception message — the RapidAPI error text is visible immediately
- **Dev aid**: `data/linkedin_raw/` — raw responses are dumped here after every successful fetch so the parser can be iterated without burning quota. Gitignored
- **Mock branch** short-circuits before identifier parsing — booth with `RAPIDAPI_KEY=mock` always shows the hardcoded author profile, no matter the input

### Phase 4.3 — Opener prompt + LinkedIn avatar plumbing (`5eb9970`)
**Prompt**: after the first live run, the user saw "I noticed your X — what surprised you?" An open question — but the visitor only has 3 buttons (`Tell me more` / `Skeptical, why Defy?` / `Not interested`). Rewrote `prompts/opener.txt`:
- Quote the button labels verbatim
- Forbid open-ended questions toward the visitor
- `opener_type=question` was redesigned as a *rhetorical* yes/no setup that immediately pivots into a statement
- Added good/bad shape examples in the prompt body

**Avatar**: added `Profile.avatar_url: str | None`, column `profiles.avatar_url Text` + an idempotent ALTER-TABLE migration in `db.init_db()` (catch duplicate-column-name, no Alembic). The parser picks 200×200 → 400 → 800 → 100 → first from `data.avatar[]`. The frontend puts `<img class="profile-portrait-img">` into the existing portrait box, the real image reveals only after `onload` (silent fallback to placeholder if the signed URL 404s). CSS `filter: grayscale(1) contrast(1.05)` to fit the documentary B&W panel style.

### Phase 4.4 — LLM-driven continuations + visible logging (`d70aeda`)
The user: "after two Tell me more the dialogue stops producing new lines; I see nothing in the logs". **Two causes**:

1. **Continuation turns were not going through the LLM at all** — `templates.render_continuation` had 5 hardcoded templates (per tone), repetition inevitable. The templates also contained open questions (`"What signal would tell you this isn't that?"`). Rewrote `_produce_continuation` via LLM with a new `prompts/continuation.txt`: gives the LLM the full dialogue history (format `Agent (turn N): ... / Visitor: [Tell me more.]`), requires no-repetition, branches on `last_choice` (positive → advance with a new detail, skeptical → defuse with a concrete reason about Defy, negative → graceful close). Templates remain as a fallback for offline LLM but are rewritten into statements / narrow yes-no — no open questions
2. **Logging was not configured** — there was no `basicConfig` anywhere, so `log.info` from modules silently dropped into the void. Added `logging.basicConfig(level=INFO, format=...)` in `app.py` plus per-call logs in `llm/client.py` (`LLM call → ... (prompt=N chars)` + `LLM call ✓/✗ ... in Nms`)

### Phase 4.4.1 — Verbose diagnostic logging (`3491146`)

- `take_turn` logs every turn (id/turn#/choice/interest/delta) and every next step (id/turn#/reply preview)

## Key architectural decisions

- **No SDK for LLM** — all three providers (Ollama/Anthropic/OpenAI) go through `httpx` directly (TASK.md §3). OpenAI streaming is SSE with a `data: [DONE]` sentinel + `delta.content` fields
- **Hybrid identifier input** — normalisation happens in one place (`_username_from_input`), internally always the canonical URL form. If we swap to a provider that wants URLs tomorrow, the frontend changes nothing
- **Mock branch before parsing** — `RAPIDAPI_KEY=mock` short-circuits *before* extracting the handle, because the booth demo must show a stable example profile regardless of the input
- **Disk cache stores the raw API response, not the Profile object** — this is what let us fix the parser (Phase 4.2 wrap bug) and automatically get an updated Profile on the next cache read, without re-fetch
- **Idempotent migration in init_db** — no Alembic, just `try: ALTER TABLE ADD COLUMN; except: pass`. Acceptable while schema changes are rare and single-dev
- **Templates as fallback, not main path** — Phase 1B had it the other way around. Now the LLM is the default, templates only when the LLM is offline

## Open questions / next steps

See `current priorities.md`. Short form:
1. **Continuation diversity** — all LLM responses ride a single rail ("another credential": partnership, cohort, case studies, scoping call). Need rotation across response types (question → personal → story → call to action) via explicit categories in the prompt
2. **Scenario testing** — so far we tested only positive×5. Run real skeptical / negative / mixed branches, look at response quality. Check the graceful close on negative
3. **Avatar caching strategy** — signed URLs from LinkedIn live a few months then 404. The cache currently keeps the URL forever. Either parse the `e=` query param and invalidate the cache at expiry, or proxy avatars through the backend
4. **Profile classification for live** — `cluster_id: —` in the right panel for live profiles because embedding classification is not implemented. This blocks persona-based rules for real LinkedIn visitors (see below)
5. **2-level clustering (deferred user discussion)** — we cluster episode summaries for induction, but search for the profile cluster with the same embedding. May need a separate profile embedding for live classification

## What I learned new about the ecosystem

- **RapidAPI providers are very unstable** — two providers sunset within a week. Disk cache + raw dump = insurance against another parser rewrite
- **OpenAI gpt-4o-mini is good for continuation** — 2-3s latency, ~$0.0001/turn, naturally varies its responses
- **PowerShell quoting** — for curl you need the `--%` stop-parsing token or Invoke-RestMethod with a hashtable → ConvertTo-Json. JSON in single quotes does not work in PS
- **Windows console encoding** — my `→` / `✓` / `✗` render as `→` / `✓` / `✗` in PowerShell logs, but that is a decode-side artifact, not a bug
- **`python-dotenv` strips quotes** automatically, but multi-line strings (Python tuple style) — not, must be on one line
